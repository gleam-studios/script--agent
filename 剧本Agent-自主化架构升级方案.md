# 剧本 Agent · 自主化架构升级方案

> **文档性质**：工程升级蓝图（规划文档）  
> **适用项目**：本仓库「海外女性向 BL 短剧」剧本 Agent（Next.js Web + Prompt 包 + 项目存储）  
> **版本**：v1.0  
> **说明**：本文仅描述目标架构、阶段划分与交付物，不包含具体代码实现。

---

## 1. 背景与目标

### 1.1 背景

当前系统以**超长系统提示词 + 人机多轮对话**为主：将 `agent/prompts/`、`agent/templates/` 等静态内容拼接进单次 `chat/completions` 请求，依赖模型自觉遵循 `flowchart` 中的 STAGE 1–5 纪律。工程侧提供阶段推断（`stage-detect`）、产物解析（`artifact-extract`）与项目持久化，但**没有**服务端工具调用闭环、**没有**显式可执行状态机、**没有**按查询动态检索的知识注入。

业界「自主型 Agent」强调：**模型负责推理与决策，工具负责真实动作，状态与流程由系统约束，必要时配合记忆、反思与 RAG**。本方案旨在在不改变产品主线（编剧辅助、分阶段交付）的前提下，将现有能力**渐进升级**为可维护、可测试、可扩展的自主化架构。

### 1.2 升级目标（可验收）


| 维度        | 目标描述                                                                   |
| --------- | ---------------------------------------------------------------------- |
| **可控流程**  | STAGE 推进、回退、暂停等待确认等由**服务端状态**约束，而非仅依赖模型自觉                              |
| **可执行动作** | 通过 **Function Calling / Tools** 读写项目产物、加载模板片段、检索素材，参数与描述遵循 JSON Schema |
| **闭环执行**  | 支持单次用户意图触发**多步**「推理 → 工具 → 观察 → 再推理」，直至交付或显式停止条件                       |
| **质量内环**  | 关键节点具备 **Validate / Reflect**（规则 + 可选二次 LLM），未通过则不盲目推进阶段               |
| **知识策略**  | 规则与模板：**静态或按需加载**；长篇网文与外部资料：**动态 RAG**（检索后再生成）                         |
| **可运维**   | 日志、指标、版本与回滚策略可随阶段引入，最终与 MLOps 治理对齐                                     |


### 1.3 非目标（本阶段不承诺）

- 替换现有 UI 设计语言或重写全部前端交互（可在后续迭代按需调整）。
- 一次性迁移至某一特定云厂商或绑定单一 LLM 供应商。
- 在未验证检索质量前，承诺「零幻觉」或完全自动化无需人工确认。

## 2. 现状摘要（基线）

### 2.1 已有资产

- **Prompt 体系**：`main_prompt`、`role`、`rule`、`skill`、`flowchart`、子 Agent 角色文件；模板目录结构化程度高，与 STAGE 强相关。
- **Web 层**：OpenAI 兼容 API 流式对话；系统提示通过 `prompt-loader` 聚合。
- **解析与展示**：`stage-detect`、`artifact-extract`、项目侧边栏与产物卡片。
- **存储**：项目 JSON（含消息与产物等，以实际 `project-store` 为准）。

### 2.2 主要缺口

- API 层无 **tools** 定义与执行器。
- 无服务端 **多轮 agent 循环**（每步 observation 写回状态）。
- STAGE 为**软约束**（Prompt），非**硬状态**（代码 + 数据）。
- 上下文以**全量静态拼接**为主，缺少**按任务检索**的 RAG 管道。
- 无独立的 **Validate / Reflect** 节点与可回归的测试用例集合。

## 3. 目标架构总览

### 3.1 分层模型（与业界框架对齐）

```
┌─────────────────────────────────────────────────────────────┐
│  表现层：对话 UI、阶段指示、产物面板、进度/审计展示              │
├─────────────────────────────────────────────────────────────┤
│  编排层：状态机 / 有向图（可选 LangGraph 或自研轻量图）         │
│         节点：Plan / Draft / Tool / Observe / Validate /      │
│              Reflect / HumanGate / Finalize                  │
├─────────────────────────────────────────────────────────────┤
│  Agent 运行时：ReAct 或 PAOR 循环、工具路由、最大步数/超时        │
├─────────────────────────────────────────────────────────────┤
│  工具层：项目读写、模板片段、检索、（可选）外部 API              │
├─────────────────────────────────────────────────────────────┤
│  模型层：主模型 +（可选）小模型专用于校验、摘要、分类            │
├─────────────────────────────────────────────────────────────┤
│  数据层：项目存储、向量库、对象存储（网文原文）、日志与指标       │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 核心原则

1. **流程框架在代码，智能判断在模型**：哪些阶段必须确认、哪些工具可调用，由编排层保证。
2. **单一事实源**：已确认的剧情节点、人物设定等以结构化存储为准，模型通过工具读取，避免长对话漂移。
3. **内外分层**：内部 Thought / 原始工具 JSON 可记日志；对用户仅展示步骤级进度与必要说明。
4. **渐进演进**：优先在现有 Next API 同仓内增加 `agent` 或扩展 `chat`，避免大爆炸重写。

## 4. 状态与数据设计

### 4.1 建议的服务端状态结构（概念）

以下字段为逻辑模型，实际实现时可映射到项目 JSON 或独立 `agent_state` 表/文件。


| 字段                  | 说明                                                               |
| ------------------- | ---------------------------------------------------------------- |
| `current_stage`     | 1–5 或 `intake` / `clarify` 等扩展                                   |
| `stage_status`      | 如 `drafting` / `awaiting_user_confirm` / `validated` / `blocked` |
| `plan`              | 当前任务拆解（要点列表，可由模型生成后由系统裁剪）                                        |
| `artifacts`         | 与现有 `Artifact` 对齐，带版本或 `updatedAt`                               |
| `open_questions`    | 待用户补充的信息                                                         |
| `tool_trace`        | 每步 tool 名、参数摘要、结果摘要（可脱敏）                                         |
| `validation_errors` | 最近一次校验失败原因列表                                                     |
| `reflect_notes`     | 反思产出的可执行调整项（短句规则）                                                |


### 4.2 与用户消息的边界

- **用户消息**：自然语言指令、素材粘贴、明确「确认 / 回退 / 仅改第 N 集」等。
- **系统事件**：由前端或编排层写入（例如「用户点击确认进入 STAGE 3」），避免模型伪造阶段跳转。

## 5. 工具（Tools）设计纲要

### 5.1 设计规范

- 每个工具：**名称、一句话描述、JSON Schema 参数、返回值结构**（含错误码）。
- 禁止模糊描述（如「处理一下文档」）；必须写清路径/ID 与副作用（只读 vs 写入）。
- 工具实现与模型供应商解耦：统一内部 `ToolExecutor` 接口。

### 5.2 建议的首批工具集（MVP）


| 工具名（示例）                  | 职责                                            |
| ------------------------ | --------------------------------------------- |
| `get_project_summary`    | 返回当前阶段、已确认产物列表、开放问题                           |
| `get_artifact`           | 按 `stage` + `subKey` 读取已存储片段                  |
| `upsert_artifact`        | 在用户或系统确认后写入/更新产物（带版本策略）                       |
| `load_template_section`  | 按模板名与章节 ID 返回 Markdown 片段，替代全量 templates 进上下文 |
| `search_project_context` | 在当前项目消息与产物中关键词/语义检索（后期可接向量）                   |
| `set_stage`              | **仅编排层或受信路径**可调用，用于确认后推进/回退（防模型滥用需权限设计）       |


### 5.3 扩展工具（第二阶段）

- `ingest_document` / `search_corpus`：网文章节、PDF、参考资料的切片与 RAG 检索。
- `run_lint_rules`：对白 AI 味、必备字段等规则扫描（可接子 Agent 专精提示）。
- `export_bundle`：导出给下游的打包格式（可选）。

### 5.4 权限与安全

- 区分 **模型可调工具** 与 **仅服务端内部调用**（如直接改 `current_stage`）。
- 对写入类工具：默认要求 `awaiting_user_confirm` 或显式用户指令触发。
- 日志中可对 API Key、原文版权敏感段落脱敏。

## 6. 编排与循环：ReAct / PAOR 落地方式

### 6.1 运行时循环（伪流程）

1. 组装 `state` + 最近对话 + 当前阶段系统指令。
2. 调用 LLM（支持 tools）；若返回 tool_calls，则执行工具，将结果以 **tool** 消息写回，**步数 +1**。
3. 若返回最终自然语言且含结构化输出，进入 **Validate**。
4. Validate 失败 → **Reflect**（生成修正策略）→ 回到步骤 2 或要求用户补充。
5. 达到 `max_steps` 或超时 → 优雅降级：返回当前草稿 + 未完成原因。

### 6.2 与流式前端的兼容策略

- **方案 A**：保留现有 SSE 流式，仅展示「最终 assistant 文本」；工具与思考在后台执行完毕后一次性输出（实现简单，工具多步时用户等待较长）。
- **方案 B**：扩展事件类型（如 `tool_start` / `tool_end` / `stage_change`），前端展示进度条（体验更好，改动较大）。
- **建议**：MVP 采用方案 A；第二阶段实施方案 B。

### 6.3 LangGraph（或同类）引入时机

- **触发条件**：分支超过 3 类（成功 / 缺信息 / 校验失败 / 用户回退 / 人工审核），且条件边开始难以在单文件维护。
- **收益**：可视化调试、检查点、人机暂停节点标准化。
- **风险**：团队学习成本、与 Next.js 部署模型的集成复杂度。

**建议路径**：Phase 2 末评估；若自研状态机已超过 ~300 行且分支爆炸，则引入 LangGraph 迁移编排层。

## 7. RAG 策略（剧本场景）

### 7.1 内容分类


| 类型                | 建议策略                             |
| ----------------- | -------------------------------- |
| 规则、红线、输出格式        | 静态或 `load_template_section` 按需加载 |
| 已生成并经确认的 STAGE 产物 | 工具 `get_artifact` + 必要时嵌入向量索引    |
| 长篇网文原文、外部研报       | 分块、嵌入、混合检索（关键词 + 向量），引用片段写入上下文   |
| 对话历史              | 摘要压缩 + 检索；避免无限增长                 |


### 7.2 质量要点

- 引用与页码/章节号一并返回，便于核对。
- 多片段冲突时：在 Prompt 中要求并列展示来源，禁止强行合并为单一数字事实（与行业报告 Agent 同理）。

## 8. Validate / Reflect 设计

### 8.1 Validate（校验）

- **规则引擎**：基于正则/AST 的检查（如是否含「集尾卡点」、STAGE 5 是否混用开发版与交付版字段）。
- **模式校验**：对 JSON 化中间结构（若引入）做 Schema 校验。
- **LLM 裁判（可选）**：低成本模型对照 `rule.md` 做 checklist 打分，输出 `pass` / `fail` + 理由。

### 8.2 Reflect（反思）

- 输入：Validate 错误、最近 tool_trace、当前 stage。
- 输出：短列表「下一步可执行调整」（改检索词、补采访用户、重写某节），由编排层决定是自动重试还是转用户。

## 9. 分阶段路线图

### Phase 0：准备（1 周内可完成，视人力而定）

- 梳理现有 `project-store` 与 `Artifact` 类型，确认**唯一键**与合并策略。
- 定义 `agent_state` 与项目文件的迁移方案（向后兼容旧项目）。
- 建立**工具清单与 JSON Schema** 文档（本文第 5 节细化版）。

**交付物**：工具契约表、状态字段说明、旧数据兼容说明。

---

### Phase 1：工具化 + 受控多步（MVP）

- 新增 API 路由（例如 `/api/agent` 或与 `chat` 并存），实现 **Function Calling** 与 **ToolExecutor**。
- 实现首批只读工具：`get_project_summary`、`get_artifact`、`load_template_section`。
- 实现受控写入：`upsert_artifact`（带确认闸门）。
- 服务端 **max_steps**、超时、错误处理；内部 trace 落日志。
- 系统提示词瘦身：从「全量 templates」改为「按需工具加载」为主路径。

**交付物**：可演示「一次请求内多工具调用」的最小 Demo；更新部署说明。

**验收标准**：同一项目内，模型能先读已有产物再续写，且不在上下文中重复塞入全部模板。

---

### Phase 2：硬状态机 + 人机门闸

- `current_stage` 与 `stage_status` 由服务端维护；用户「确认」通过显式事件触发 `set_stage`（或等价内部 API）。
- 增加节点：**HumanGate**（等待确认）、**回退到上一 STAGE** 的数据一致性（产物版本或快照）。
- 前端可选：确认按钮与阶段锁定提示文案。

**交付物**：状态转换图（文档 + 单测覆盖主要路径）。

**验收标准**：模型无法仅凭自然语言绕过未确认阶段（在系统实现层面禁止，而非仅 Prompt）。

---

### Phase 3：Validate / Reflect 内环

- 接入规则校验 + 可选二次 LLM 校验。
- 失败自动重试次数上限；超过则向用户输出清晰缺失项。

**交付物**：校验规则列表与样例项目集（黄金用例）。

**验收标准**：对已知「坏输出」样例，系统能稳定拦截或触发重写。

---

### Phase 4：RAG 与大规模素材

- 网文/文档 ingestion 管道、向量库选型、引用策略。
- `search_corpus` 工具与引用展示。

**交付物**：检索评测集（Recall@k、人工相关性抽样）、成本估算。

---

### Phase 5：编排升级与运维（可选 LangGraph + MLOps）

- 引入 LangGraph（或等价）重构编排层。
- 监控：工具失败率、每任务步数、延迟、Token 成本、校验失败率。
- 模型版本灰度、配置中心、告警与回滚。

**交付物**：运维仪表盘指标定义、上线 Runbook。

---

## 10. 测试与质量保障

- **单元测试**：工具参数校验、artifact 合并、`stage-detect` 与编排状态机转换。
- **集成测试**：Mock LLM 返回固定 tool_calls，验证循环与落库。
- **黄金场景**：从灵感 → STAGE 5 样例项目的端到端脚本（可夜间跑）。
- **回归 Prompt**：重大 Prompt 变更需对比黄金场景输出 diff。

---

## 11. 风险与应对


| 风险            | 应对                              |
| ------------- | ------------------------------- |
| 模型不遵守 tool 格式 | 严格 Schema + 重试 + 降级为纯文本提示用户     |
| 工具调用死循环       | max_steps、重复检测（相同 tool+args）、熔断 |
| 上下文仍过长        | 模板按需加载 + 对话摘要 + RAG Top-k       |
| 用户反感等待        | Phase 1 明确预期；Phase 2+ 进度事件流     |
| 版权与隐私         | 网文与项目数据加密、访问控制、日志脱敏             |


---

## 12. 成功指标（KPI 建议）

- **阶段误推进率**：应趋近于 0（由硬状态机衡量）。
- **单任务平均 Tool 步数**：控制在合理区间（避免浪费 Token）。
- **校验一次通过率**：随 Reflect 迭代上升。
- **用户为「补全信息」发的轮次**：下降（open_questions 机制生效）。
- **P95 响应时间**：在工具化后仍可接受（需并行与缓存优化）。
